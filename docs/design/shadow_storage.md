# 影子存储 (Shadow Storage) 详细设计说明书

## 1. 概述
影子存储是 Illacme-plenipes 引擎实现“工业级同步”的核心基座。它通过在本地维护一个受控的、不可变的翻译镜像层，解决了 AI 算力冗余消耗、物理资产状态自愈以及增量同步的原子化问题。

**版本状态**：v6.0.1 (Stage V6 Delta Block Engine 适配版)
**物理路径**：`{vault_root}/.illacme-shadow/`

---

## 2. 核心架构设计

### 2.1 双层存储结构
影子存储采用“全量镜像 + 碎片指纹”的双层拓扑结构：

1.  **全量镜像层 (`/en/`, `/zh-hans/`, etc.)**
    *   **结构**：严格映射源文档库的目录树。
    *   **职责**：存储最近一次同步成功的完整译文产物。
    *   **用途**：用于全文件哈希校验（Fast Path）和物理资产分发。

2.  **语义碎片层 (`/blocks/`)**
    *   **结构**：`blocks/{lang_code}/{md5_hash}.txt`
    *   **职责**：存储经过语义切片（Callout, Header, Paragraph）后的原子级翻译结果。
    *   **用途**：实现 Stage V6 的 Delta Block 增量逻辑，支持跨文件的翻译复用。

### 2.2 逻辑寻址协议
引擎通过以下三个维度的交集定位影子资产：
*   **Source Hash**：源文件内容的全局 MD5。若一致，则触发“全速回滚”。
*   **Block Fingerprint**：语义块内容的 MD5 + 类型标识。若匹配，则触发“原子复用”。
*   **Slug Integrity**：验证文件名与影子路径的一致性，防止因重命名导致的“资产漂移”。

## 3. 关键工作流

### 3.1 增量同步流 (Delta Sync)
1.  **切片**：`MarkdownBlockParser` 将源文件切分为 $N$ 个语义块。
2.  **寻址**：遍历 $N$ 个块，计算指纹并在 `blocks/` 中查询。
3.  **命中处理**：
    *   **HIT**：直接从 `.txt` 碎片中读取译文。
    *   **MISS**：将该块发送至 AI 适配器，获取译文后立即存入碎片层。
4.  **组装**：将所有译文块（HIT + NEW）合并，更新至全量镜像层。

### 3.2 幽灵清洗流 (Ghost Cleaning)
1.  **扫描**：`Janitor` 服务扫描 SSG 前端目录（如 `i18n/`）。
2.  **比对**：将前端文件与影子存储中的“白名单”进行对齐。
3.  **隔离**：若前端存在文件但在影子存储中无记录，判定为“幽灵节点”，执行物理切除。

## 4. 治理与安全规约

### 4.1 数据一致性 (AEL-Iter-ID)
*   影子存储中的哈希值必须与 `.plenipes/ledger.json` (账本) 严格对齐。
*   任何手动的影子存储修改都可能导致下一次同步时触发 `DEGRADED` (降级模式)。

### 4.2 算力保护协议
*   **零冗余原则**：只要影子碎片存在，系统严禁发起重复的 AI 请求。
*   **强制自愈**：若前端产物丢失，系统应优先从影子存储恢复，而非重新翻译。

### 4.3 物理清理
*   影子存储受 `.gitignore` 保护（建议），不应提交至代码仓库，以保持文档库的纯净度，但支持在多设备间手动迁移。
